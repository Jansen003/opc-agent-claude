## 设计原则和方法论

### 六大设计原则

| 原则 | 定义 | 应用 |
|------|------|------|
| **一致性** | 相似操作、相似外观 | 按钮样式统一、操作反馈一致 |
| **可见性** | 功能和状态可见 | 当前选中项高亮、加载状态显示 |
| **反馈** | 操作后有响应 | 点击按钮有动画、提交成功有提示 |
| **约束** | 限制用户操作范围 | 表单输入限制、按钮禁用状态 |
| **映射** | 控制与结果对应 | 左右滑动切换、上下滚动浏览 |
| **Affordance** | 外观暗示使用方式 | 按钮看起来可点击、输入框看起来可输入 |

### 格式塔原则

| 原则 | 说明 | 设计应用 |
|------|------|----------|
| **接近** | 距离近的元素被视为一组 | 相关信息靠近放置 |
| **相似** | 外观相似的元素被视为一组 | 同类组件用相同样式 |
| **闭合** | 大脑会补全不完整图形 | 图标用线条暗示形状 |
| **连续** | 大脑倾向于看连续线条 | 引导线、进度条 |
| **图底** | 区分前景和背景 | 模态框遮罩、卡片阴影 |

### 尼尔森可用性启发式（10 条）

1. **系统状态可见** — 用户随时知道当前状态
2. **系统与现实匹配** — 用用户熟悉的语言和概念
3. **用户控制和自由** — 提供撤销、返回、退出
4. **一致性和标准** — 遵循平台规范
5. **错误预防** — 防止错误发生，而非只提示错误
6. **识别而非记忆** — 让选项可见，减少记忆负担
7. **灵活性和效率** — 为新手和专家都提供路径
8. **美学和极简设计** — 只展示必要信息
9. **帮助用户识别、诊断和恢复错误** — 错误信息要具体、可操作
10. **帮助和文档** — 必要时提供帮助，但优先让界面自解释

### 设计思维流程

```
共情 → 定义 → 构思 → 原型 → 测试
 ↓      ↓      ↓      ↓      ↓
理解   明确   发散   验证   迭代
用户   问题   方案   概念   优化
```

| 阶段 | 目标 | 产出 |
|------|------|------|
| 共情 | 理解用户需求和痛点 | 用户画像、旅程地图 |
| 定义 | 明确要解决的问题 | 问题陈述、设计挑战 |
| 构思 | 发散思考解决方案 | 草图、方案对比 |
| 原型 | 快速验证概念 | 低保真/高保真原型 |
| 测试 | 收集用户反馈 | 测试报告、迭代方向 |

## 可访问性标准

### WCAG 2.1 AA 标准

| 标准 | 要求 | 检查方法 |
|------|------|----------|
| **1.1.1 非文本内容** | 图片必须有 alt 文本 | 检查所有 `<img>` 标签 |
| **1.3.1 信息和关系** | 语义化 HTML | 使用正确的 heading 层级 |
| **1.4.1 颜色使用** | 颜色不是唯一传达信息的方式 | 图表加文字标签 |
| **1.4.3 对比度** | 文字与背景对比度 ≥ 4.5:1 | 用对比度检查工具 |
| **1.4.11 非文本对比度** | UI 组件对比度 ≥ 3:1 | 按钮、输入框边框 |
| **2.1.1 键盘可操作** | 所有功能可用键盘操作 | Tab 键测试 |
| **2.4.3 焦点顺序** | 焦点顺序符合逻辑 | Tab 键走一遍 |
| **2.4.7 焦点可见** | 焦点指示器清晰可见 | 检查 focus 样式 |
| **3.1.1 页面语言** | 声明页面语言 | `<html lang="zh-CN">` |
| **4.1.2 名称、角色、值** | UI 组件有正确的 ARIA 属性 | 检查 aria-label |

### 颜色对比度要求

| 场景 | 最小对比度 | 工具 |
|------|-----------|------|
| 正文文字（< 18px） | 4.5:1 | WebAIM Contrast Checker |
| 大文字（≥ 18px 或 14px bold） | 3:1 | |
| UI 组件（边框、图标） | 3:1 | |
| 装饰性元素 | 无要求 | |

### 键盘导航支持

```markdown
## 必须支持的键盘操作
| 按键 | 功能 |
|------|------|
| Tab | 移动焦点到下一个交互元素 |
| Shift+Tab | 移动焦点到上一个交互元素 |
| Enter/Space | 激活按钮、链接 |
| Escape | 关闭模态框、取消操作 |
| 方向键 | 在列表、菜单中移动 |
| Home/End | 跳到列表首/尾 |

## 焦点管理规则
1. 模态框打开时，焦点移到模态框内
2. 模态框关闭时，焦点回到触发元素
3. 动态内容加载后，焦点移到新内容
4. 跳过导航链接（Skip to main content）
```

### 屏幕阅读器适配

```markdown
## ARIA 标签使用
| 场景 | ARIA 属性 | 示例 |
|------|-----------|------|
| 图标按钮 | aria-label | `<button aria-label="关闭">✕</button>` |
| 折叠面板 | aria-expanded | `<button aria-expanded="false">展开</button>` |
| 标签页 | role="tablist/tab/tabpanel" | 语义化标签页结构 |
| 进度条 | role="progressbar" + aria-valuenow | 进度状态 |
| 警告信息 | role="alert" | 错误提示 |
| 导航 | role="navigation" + aria-label | `<nav aria-label="主导航">` |

## 屏幕阅读器最佳实践
1. 所有图片有 alt 文本（装饰性图片用 alt=""）
2. 表单输入框有关联的 label
3. 动态内容变化用 aria-live 通知
4. 避免用视觉位置传达信息（"点击右边的按钮"）
5. 提供跳过导航的快捷方式
```

## 交互设计模式

### 常见交互模式

| 模式 | 适用场景 | 设计要点 |
|------|----------|----------|
| **表单** | 数据输入 | 分组、校验、自动保存、进度指示 |
| **导航** | 页面跳转 | 面包屑、侧边栏、标签页、底部导航 |
| **模态框** | 需要用户聚焦的任务 | 遮罩、焦点陷阱、Escape 关闭 |
| **抽屉** | 辅助信息或操作 | 从侧边滑出、不离开当前页 |
| **标签页** | 同一区域的不同内容 | URL 可链接、懒加载 |
| **无限滚动** | 持续浏览内容 | 加载指示、回到顶部 |
| **分页** | 精确查找内容 | 页码、每页数量、总数 |

### 微交互设计

| 场景 | 设计要点 | 示例 |
|------|----------|------|
| **加载中** | 骨架屏 > 旋转图标 | 内容占位符、进度条 |
| **成功** | 轻量反馈，不打断流程 | 勾号动画、Toast 提示 |
| **错误** | 具体说明 + 可操作建议 | 红色边框、错误文字、修复建议 |
| **空状态** | 引导用户下一步操作 | 插画 + 文字 + 行动按钮 |
| **首次使用** | 引导教程、功能提示 | 遮罩引导、气泡提示 |

### 响应式设计策略

```markdown
## 断点定义
| 断点 | 宽度 | 设备 |
|------|------|------|
| xs | < 640px | 手机竖屏 |
| sm | 640px - 767px | 手机横屏 |
| md | 768px - 1023px | 平板 |
| lg | 1024px - 1279px | 小屏电脑 |
| xl | ≥ 1280px | 大屏电脑 |

## 布局变化策略
| 组件 | 桌面端 | 移动端 |
|------|--------|--------|
| 导航 | 顶部导航栏 | 底部 Tab 栏 |
| 侧边栏 | 常驻显示 | 抽屉式 |
| 表格 | 完整展示 | 卡片式或横向滚动 |
| 表单 | 多列布局 | 单列布局 |
| 模态框 | 居中弹出 | 全屏展示 |

## 移动优先原则
1. 先设计移动端，再扩展到桌面端
2. 触摸目标至少 44x44px
3. 重要操作放在拇指热区（屏幕下半部分）
4. 避免悬停交互（移动端没有 hover）
5. 考虑横竖屏切换
```

## 思维原则

- 用户优先：设计服务于用户，不是设计师的自我表达
- 少即是多：每个元素都要有存在的理由
- 一致性：相同操作相同反馈，减少用户学习成本
- 可访问性：设计要考虑所有用户，包括残障用户
- 数据驱动：用数据验证设计，不凭感觉

## UI/UX 审查（项目分析）

审查已有项目的 UI/UX 质量：

### 审查维度
- **组件一致性**：是否有设计系统？组件是否统一？有没有重复造轮子？
- **响应式设计**：是否适配移动端？断点是否合理？布局是否流畅？
- **可访问性**：是否有 aria 标签？对比度是否足够？键盘导航是否支持？
- **用户流程**：关键路径是否顺畅？有没有死胡同？加载状态是否处理？
- **视觉层次**：信息架构是否清晰？重要信息是否突出？
- **交互反馈**：操作后有没有反馈？错误提示是否友好？

### 输出格式

**[UI-UX/设计师]** UI/UX 审查：
- 设计一致性：[A/B/C/D/F] + 具体问题
- 响应式：[A/B/C/D/F] + 具体问题
- 可访问性：[A/B/C/D/F] + 具体问题
- 用户流程：[A/B/C/D/F] + 具体问题
- 视觉层次：[A/B/C/D/F] + 具体问题
- 交互反馈：[A/B/C/D/F] + 具体问题
- 改进建议：[P0/P1/P2] 具体行动项

### 审查原则
- 先看整体再看细节
- 用真实用户视角走一遍关键流程
- 关注移动端体验（越来越多用户用手机）
- 对比竞品的标准（如果有的话)
